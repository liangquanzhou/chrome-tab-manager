# CTM — Change Policy

需求变更是正常的。混乱的变更才是问题。本文件定义变更流程，面向 agent。

## 1. 变更顺序

Agent 必须按此顺序处理变更：

1. **澄清**请求的变更
2. **分类**变更类型（见 §2）
3. **先更新**对应的文档
4. **记录或修订**决策（`01_PRODUCT.md` Decision Log）
5. **然后**修改实现
6. **更新**验收标准
7. **汇报**残余影响

对重大变更，此顺序是强制的。

## 2. 变更分类

每个变更请求必须先归入以下类别之一：

### 2.1 产品范围变更

例：新增顶层能力域、改变产品定义、改变某个领域是否一等公民。

影响最大的变更类型。必须先更新：
- `01_PRODUCT.md`
- `02_DOMAIN.md`

### 2.2 领域模型变更

例：新增核心对象、拆分对象、改变资源所有权。

必须先更新：
- `02_DOMAIN.md`
- `09_DESIGN.md`

### 2.3 用户体验变更

例：新增/移除顶层区域、改变导航模型、在 CLI/TUI/Palette 间迁移操作。

必须先更新：
- `04_INTERACTION.md`
- `10_TUI.md`（如涉及 TUI）

### 2.4 架构/契约变更

例：新增 action、改变消息语义、新增运行时模块。

必须先更新：
- `09_DESIGN.md`
- `12_CONTRACTS.md`

### 2.5 交付/阶段变更

例：调整 build 顺序、改变实现优先级、重新定义完成里程碑。

必须先更新：
- `03_CAPABILITIES.md`（build order）
- `11_PLAN.md`
- `13_ACCEPTANCE.md`

## 3. 决策日志规则

如果变更影响产品方向，必须更新 `01_PRODUCT.md` 的 Decision Log。

新决策替代旧决策时，必须：
- 明确写出替代了哪个决策
- 解释为什么改变
- 解释新的影响

不可以悄悄重写旧的含义。

## 4. 变更安全检查

实施重大变更前，agent 必须回答：

1. 是否改变了产品范围？
2. 是否改变了领域模型？
3. 是否改变了用户导航？
4. 是否改变了 command surface 职责分工？
5. 是否改变了 sync 行为？
6. 是否改变了验收标准？
7. 这次改变让未来变更更容易还是更难？

如果让未来变更更难，agent 必须显式说明。

## 5. 可逆与不可逆变更

### 可逆（成本低）

- 命名改进
- UI 位置调整
- 新增命令
- 新增视图

### 不可逆（成本高）

- 对象身份变更
- Sync 模型变更
- Workspace 语义变更
- Source of Truth 变更
- Contract shape 变更

高成本变更需要先做更多文档工作。

## 6. "看起来小但实际是结构性变更"

有些变更听起来小但实际改变模型：

- "能不能给书签加 tag"
- "能不能也同步这个"
- "能不能给 workspace 加个行为"

遇到这种情况，agent 必须：

1. 解释这是模型级变更
2. 先更新文档
3. 然后实现

防止隐性架构漂移。

## 7. 真正的小变更

如果变更不涉及产品范围、领域模型、导航、契约、sync 行为，agent 可以直接实现，只需要在最近的相关文档中更新即可。

不是每个小变更都需要走完整流程。

## 8. 非技术 Owner 规则

因为 owner 不读代码，agent 不能依赖：

- 隐含的架构知识
- "从实现中很明显"
- 代码阅读作为主要解释手段

agent 必须通过文档让变更可理解。这是 owner 在不编码的情况下保持控制的方式。

## 9. 产品连贯性标准

变更的最重要标准不是：

- 编码快不快

而是：

- 产品是否仍然连贯
- 未来扩展是否仍然可能

这是防止"后面做完了要改，那我就抓瞎"的核心保护。
