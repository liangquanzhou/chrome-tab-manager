# Codex Guide V2

## 目的

这份文档不是局部 code review，也不是功能盘点。

它的目标是从**顶层**重新审视当前 CTM 的状态，给出下一阶段的工程与产品判断：

1. 这个项目现在到底是什么
2. 它当前最强的部分是什么
3. 它下一阶段最危险的偏移是什么
4. 应该如何定义接下来一轮迭代的总目标、边界和验收门槛

这版相对 `codex-guide-v1.md`，更强调：

- 项目定位
- 阶段边界
- 取舍原则
- 发布与产品面的收口标准

---

## 顶层判断

## 一句话版本

**CTM 现在已经是一个成型的浏览器工作流内核，但还不是一个完全收口的产品。**

它最强的是：

- 架构已经成立
- 主链路已经打通
- 测试基础已经足够支撑持续迭代

它最不稳的是：

- 能力广度已经开始超过产品表面的一致性
- 文档、help、CLI、TUI、distribution 没有始终保持同速收口
- 下一阶段如果继续“横向扩张”，会比前一阶段更快地产生系统分叉

所以，当前最正确的方向不是“再证明还能加多少功能”，而是：

> **把已经做出来的系统，收口成一个一致、可信、可发布、可继续扩展的产品层。**

---

## 当前项目应该如何定位

我建议把当前 CTM 的官方自我定位明确成下面这句：

> **Chrome-first、local-first、single-binary 的浏览器工作流系统**

这个定位比“browser workspace manager”更准确，也更适合当前阶段。

原因很简单：

- 它已经不仅是 tab 工具
- 也还不是一个 fully productized 的跨浏览器知识/同步平台

现在如果把它定义得太小，会低估它已经形成的系统性。  
如果把它定义得太大，会高估它当前的交付完成度。

当前最合适的定位应该同时承认两件事：

1. 它已经有系统级内核
2. 它的产品层还在 consolidation 阶段

---

## 顶层评价

## 1. 作为“系统内核”，它已经比较成熟

这是当前项目最值得肯定的地方。

### 1.1 架构形态是对的

当前核心形态已经非常清楚：

- 一个二进制 `ctm`
- 一个中心 daemon / hub
- 一个 extension bridge
- 多个入口层（CLI / TUI / install / nm-shim）
- 多个资源域（tabs / groups / sessions / collections / bookmarks / workspace / search / sync / history）

这意味着它已经不是“功能脚本集合”，而是一个有明确中心和边界的系统。

### 1.2 模块切分是可长期维护的

当前目录组织没有明显塌陷：

- 协议层、客户端层、服务层、领域层、交互层是分开的
- 本地对象模型也基本稳定
- 新功能通常可以找到自然挂载点

这让它具备了继续长期演进的条件。

### 1.3 测试质量是当前的护城河

这个项目现在最大的现实优势，不是 feature count，而是它已经有足够扎实的自动化基础。

这会直接带来两件事：

1. 可以做更快的重构和收口
2. 可以开始要求更严格的“表面契约一致性”

也就是说，现在不是“测试勉强够”，而是“测试已经足以支撑下一阶段进入 hardening mode”。

---

## 2. 作为“产品表面”，它仍然不够收口

这也是当前最主要的问题。

现在的 CTM，不是没有功能，而是**能力已经超过产品面的一致性**。

### 2.1 路由层和交互层的成熟度不对称

daemon / extension / contract 这三层在很多地方已经走得比较前。

但 CLI / TUI / help / capability 文档不总是同步。

结果就是：

- 底层可做，上层不一定自然可用
- help 里写了，不一定真实能操作
- capability matrix 看起来很全，但仍有一些“已建模/已部分接线/未形成完整用户路径”的项

这是当前最典型的“系统已成，产品未齐”信号。

### 2.2 项目开始同时维护“平台 backlog”和“产品 backlog”

这是一个重要拐点。

现在仓库里同时存在两类工作：

- 平台型工作：协议、转发、存储、镜像、sync、索引、distribution
- 产品型工作：help、keymap、视图动作、自然入口、安装路径、文档真实度

前一阶段这两类工作能混在一起推进，因为系统还在长骨架。  
现在继续混在一起推进，成本会开始上升。

如果不把这两类工作区分开，下一步迭代会很容易陷入：

- 平台继续长
- 产品面继续补一半
- review 持续发现“功能有了，但没形成完整用户路径”

### 2.3 发布面还不是第一公民

当前 CTM 具备发布骨架，但还不具备一个完全可信的 end-to-end release path。

这是顶层上必须正视的事。

一个系统如果已经走到：

- 有 `Makefile`
- 有 `.goreleaser.yml`
- 有 Homebrew 路径
- 有 `ctm install`

那“发布链路是否闭环”就不再是边角问题，而是产品可信度的一部分。

现在这块还没有完全收口，所以它更适合 internal hardening，而不是把“distribution done”当成阶段性胜利。

---

## 3. 当前最大的结构性风险，不是代码质量，而是方向漂移

从顶层看，CTM 当前真正的风险不是“代码写烂了”，而是：

> **系统已经有能力继续快速加东西，但产品层尚未建立清晰的收口纪律。**

这是个典型的第二阶段风险。

前一阶段的危险是：

- 主链路打不通
- 架构塌掉
- 测试基础不成立

这些现在都不是主要问题了。

当前阶段的危险变成：

- 功能继续增加
- 视图继续增加
- action 继续增加
- docs 继续扩
- 但“当前支持的正式路径”没有被明确限定

这会让系统逐渐进入一种状态：

- 什么都好像支持一点
- 但没有一套足够强的产品边界告诉人“什么是当前正式能力，什么是未来能力，什么是底层预留能力”

这类漂移不会立即爆炸，但会稳定提高维护成本。

---

## 当前应该明确承认的现实

我建议团队在后续文档和讨论中，把下面几件事说清楚，不再模糊表述。

## 1. 当前不是“全面产品化完成”

可以说：

- Phase 1-8 的主能力骨架已完成
- 当前处于 consolidation / closure 阶段

但不要把这个阶段理解成“可以随意扩张，剩下只是 polishing”。

现在还有几类事情并不是 polish，而是**决定下一阶段可维护性的基础工作**。

## 2. 当前也不是“只是原型”

同样不能反过来低估它。

现在已经没必要再用“先做出来再说”的心态推进，因为：

- 协议已经稳定到需要认真维护兼容性
- TUI 已经有足够多视图和快捷键，需要维护交互契约
- CLI 已经足够大，需要关心命令面的一致性
- 文档已经多到会反向影响开发

这意味着项目已经进入“有系统负债管理要求”的阶段。

## 3. 当前最重要的不是多一个 feature，而是多一个“闭环”

接下来的成功指标不该是：

- 又多了几个 action
- 又多了一个 view
- 又多了一个资源域

而应该是：

- 一个能力是否真的从底层到表面闭环
- 一个用户路径是否真能从安装走到使用
- 一个文档是否真能代表当前现实

---

## 接下来一轮迭代的总目标

我建议把下一轮的总目标定义成：

## Iteration Goal

**把 CTM 从“功能广、系统强”的工程内核，推进到“边界清晰、路径可信”的产品层。**

这句话包含 4 个子目标：

1. **路径可信**
   安装、启动、连接、使用、退出，这条主路径要有清晰且可验证的单一版本

2. **表面一致**
   CLI / TUI / help / capability matrix / contracts 之间不能长期失真

3. **语义闭合**
   multi-target、bookmark mirror、search、workspace 这几个跨层概念要说得清、行为一致

4. **继续可扩**
   为下一阶段的搜索、sync、workspace 强化预留一个更稳的底座

---

## 对下一阶段的核心建议

下面是我认为下一轮必须坚持的 6 条总原则。

## 原则 1：先做“closure backlog”，再做“expansion backlog”

建议显式把 backlog 分成两类：

### Closure backlog

- 修正表面失真
- 补齐正式用户路径
- 收口安装发布
- 收口 target / mirror / search 语义
- 更新文档和 acceptance

### Expansion backlog

- 新资源域
- 新 power features
- 更强 automation
- 多浏览器扩展
- 更复杂 sync 策略

当前阶段，closure backlog 的优先级应该整体高于 expansion backlog。

## 原则 2：明确“正式支持路径”与“底层预留能力”的区别

现在系统里已经存在一些能力：

- 底层支持
- 但没有 CLI/TUI 正式入口
- 或者有入口但没有完整交互契约

这并不一定是坏事。  
坏的是不区分这两类能力。

建议明确分三档：

1. **正式支持**
   可以写进 help、文档、发布说明、acceptance

2. **部分支持**
   底层存在，但尚未形成正式用户路径

3. **预留/实验**
   可以保留，但不应写成当前产品承诺

这条规则会显著降低“表面过强、实现半闭环”的问题。

## 原则 3：每一个 help 文案都应该被当作契约

当前 TUI/CLI 已经不是内部脚手架了。

help、keymap、status、toast 不再只是文案，而是用户对系统的**最直接 contract**。

所以后续应该建立一个简单但严格的规则：

> 只要某个动作出现在 help / keymap / capability matrix 里，它就应该被视为“被承诺的行为”。

一旦被承诺，就要满足至少其一：

- 有自动化测试
- 有明确的手工验收标准
- 有清楚的非支持说明

## 原则 4：安装/发布链路必须被当作一等能力

当前最大的顶层风险之一，是团队已经在维护 distribution 配置，但尚未把 release path 当成第一类系统路径。

这是一个阶段判断问题：

- 在早期，install 可以只是开发辅助命令
- 在现在，install / Homebrew / post_install / `--check` 已经是产品面的一部分

所以从现在起，发布链路必须满足和业务功能类似的标准：

- 可解释
- 可测试
- 可复现
- 不自相矛盾

## 原则 5：multi-target 语义必须在数据层闭环，不能只在路由层闭环

当前项目在 target routing 上已经比较成熟。  
但接下来真正决定它能不能继续扩的，不是“请求能否带 target”，而是：

- 数据缓存是否按 target 存
- 搜索是否按 target 解释
- mirror 是否按 target 归属
- workspace/defaultTarget 是否和搜索/restore 一致

也就是说：

> 只有路由层 target-aware，是不够的。  
> 数据层也必须 target-aware。

## 原则 6：文档要从“设计文档”升级为“运行中规格”

当前 `doc/` 不应该只是历史设计记录。  
它已经变成实际协作系统的一部分。

所以后续文档治理建议是：

- `12_CONTRACTS.md` 继续保持精确
- `18_CURRENT_PHASE.md` 只写当前阶段的真实状态
- `19_CAPABILITY_MATRIX.md` 维护成当前现实，不写推测
- review 文档只记录判断，不反向替代规范文档

文档越多，越要区分：

- 规范
- 现状
- review
- 愿景

## 推荐的下一轮迭代结构

我建议下一轮不要按资源域拆，而是按“收口类型”拆。

## Track A：Truth Alignment

### 目标

统一“系统现在到底支持什么”。

### 内容

- 修正 `18_CURRENT_PHASE.md` 中过时指标和强表述
- 继续把 `19_CAPABILITY_MATRIX.md` 维护成当前现实，而不是理想状态
- 为每个能力标注：
  - 正式支持
  - 部分支持
  - 预留/实验

### 成果

任何人看文档都能回答：

- 当前正式支持什么
- 哪些只是底层能力
- 哪些不是当前阶段承诺

## Track B：Surface Closure

### 目标

收口 CLI / TUI / help / keymap 的产品表面。

### 内容

- 清掉 help/keymap 与真实行为不一致的项目
- 缺 handler 的动作要么补上，要么从 help 中移除
- 选择一批最值得正式支持的路径真正打通

建议优先：

- Groups view
- Targets view
- Bookmarks 管理面
- Search / Saved Search 的表面定义

### 成果

用户看到什么，就真的能用什么。

## Track C：Data Closure

### 目标

收口最关键的数据语义问题。

### 内容

- bookmark mirror per-target
- search index 第一版
- workspace / target / search / mirror 的联动语义说明
- 对 best-effort 类动作补齐 warnings / skipped 等结果语义

### 成果

系统从“会转发请求”升级到“有明确的数据解释模型”。

## Track D：Release Closure

### 目标

把安装/发布路径变成真正可信的主路径。

### 内容

- 修复 Homebrew / `ctm install` 语义冲突
- 明确 extension ID 获取与安装路径
- 决定 install 是交互式、参数式，还是 embed-first
- 把 `install --check` 做成真正可用的 doctor-like 检查入口

### 成果

从获取二进制到跑起系统，有一条可验证、可说明、可支持的正式路径。

## Track E：Load & Scalability Hardening

### 目标

为下一阶段数据量增长和大操作增长预先控风险。

### 内容

- restore / workspace.switch 节流
- heavy-job queue
- 搜索索引维护策略
- mirror rebuild / search rebuild 的串行控制

### 成果

系统在“更多数据、更多 tabs、更多操作”时不会明显掉入性能泥潭。

## 不建议当前优先推进的方向

这一段很重要，因为顶层 review 的价值不仅在于告诉你做什么，也在于告诉你现在**不要优先做什么**。

## 1. 不优先扩浏览器矩阵

当前先把 Chrome-first 做成完整产品，比尽快扩 Brave / Edge / Arc 更值。

## 2. 不优先继续加 view

当前 TUI 已经有 9 个 view。  
接下来最重要的不是第 10 个 view，而是让现有 9 个 view 的行为、提示、帮助和入口一致。

## 3. 不优先继续加 power feature

像更复杂的 tab 管理、automation hooks、更多批处理动作，这些都应排在 closure 之后。

## 4. 不优先把 sync 做成“大承诺能力”

在数据层、冲突层、install 层都还未完全收口之前，不适合把 sync 宣传成“成熟跨设备同步能力”。

## 5. 不优先引入更重的基础设施

例如数据库。  
当前更需要的是：

- 更清晰的数据模型
- 更清晰的索引层
- 更清晰的 target 语义

不是立刻切技术栈。

## 下一阶段的成功标准

如果让我定义“guide v2 对应的下一轮成功标准”，我会用下面 5 条。

## 1. 有一条正式支持的安装路径

不是“理论上能装”，而是：

- 能解释
- 能执行
- 能检查
- 能测试

## 2. 有一套可信的产品表面

不是“功能很多”，而是：

- help 不撒谎
- keymap 不超前
- CLI/TUI 至少在核心能力上对齐

## 3. 有一套闭合的 multi-target 数据语义

不是只有请求能带 target，而是：

- mirror
- search
- workspace/defaultTarget
- 本地缓存

这些都能在 target 语义上说得清。

## 4. 有一个可持续的搜索底座

不是每次全量扫盘就算完成，而是有明确的 index / cache / rebuild 模型。

## 5. 有一个真实反映当前状态的文档系统

不是文档很多，而是关键文档真的有信号：

- 当前 phase
- capability matrix
- contract
- acceptance

这 4 类文档必须可靠。

---

## 作为 owner 的决策建议

如果从 owner 视角只保留最重要的几条，我建议是：

### 1. 把下一轮明确定义为“收口轮”，不是“扩张轮”

这会直接改变开发和 review 的优先级。

### 2. 把“正式支持”当成一个显式状态

不要再默认“代码里有 = 产品支持”。

### 3. 把 release path 纳入核心验收

不要把 distribution 看成最后再补的包装层。

### 4. 把 capability matrix 变成真正的协作工具

不是只在 review 时看，而是在开发时也用它约束“表面/底层是否一致”。

### 5. 允许系统暂时不扩功能

这个阶段最难的不是继续实现，而是忍住继续铺开的冲动。

---

## 最终结论

从顶层看，CTM 现在已经完成了最难的第一件事：

> 它已经成为一个真实存在、可运行、可演化的系统。

但它接下来要完成的第二件事，性质完全不同：

> 不是再长出更多能力，而是把已有能力收拢成一个可信的产品层。

所以我对下一阶段的总建议是：

**把 CTM 从“功能广、内核强”的工程系统，推进到“边界清晰、路径可信”的产品系统。**

一句话收尾：

**下一轮最重要的不是扩展系统，而是定义系统。**
